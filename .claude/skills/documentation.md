# Documentation

## 概要

効果的なドキュメント作成のガイドライン。APIドキュメント、アーキテクチャドキュメント、インラインコメント、READMEの標準フォーマットを定義し、プロジェクトの理解しやすさと保守性を向上させる。

## 適用場面

- 新規プロジェクトのドキュメント整備
- API仕様書の作成
- アーキテクチャ決定記録 (ADR) の作成
- ユーザー向けドキュメントの執筆

## ベストプラクティス

### 1. README標準

- プロジェクト名と一行説明を冒頭に記載する
- Quick Startセクションで5分以内に動作確認できる手順を提供する
- 前提条件（Node.jsバージョン、OS等）を明記する
- インストール・セットアップ手順をコピペ可能なコマンドで示す
- ライセンスとコントリビューションガイドへのリンクを含める

### 2. APIドキュメント

- 全てのpublic APIにJSDoc/TSDocコメントを付ける
- パラメータの型、デフォルト値、制約を明記する
- 使用例 (Example) を必ず含める
- エラーケースとその対処法を記載する
- 非推奨APIには `@deprecated` タグと移行先を記載する

### 3. アーキテクチャドキュメント

- ADR (Architecture Decision Record) で設計判断を記録する
- ADRのフォーマット: タイトル、ステータス、コンテキスト、決定、結果
- システム構成図をMermaidやPlantUMLで記述する（画像は避ける）
- モジュール間の依存関係を図示する
- 技術選定の理由と代替案の検討過程を記録する

### 4. インラインコメント

- 「何をしているか」ではなく「なぜそうしているか」をコメントする
- 複雑なアルゴリズムには参考リンクを含める
- TODO/FIXME/HACKにはIssue番号を紐付ける: `// TODO(#123): refactor`
- マジックナンバーには名前付き定数と説明コメントを付ける
- 不要になったコメントアウトコードは削除する

### 5. ドキュメントのメンテナンス

- コードと同じPRでドキュメントを更新する
- 古いドキュメントは削除するか、アーカイブする
- CIでドキュメントのリンク切れを検出する
- ドキュメントもレビュー対象に含める
- バージョニングが必要なドキュメントにはlast-updatedを記載する

## チェックリスト

- [ ] READMEにQuick Startセクションがある
- [ ] public APIにTSDoc/JSDocが付いている
- [ ] アーキテクチャ判断がADRで記録されている
- [ ] Mermaid/PlantUMLで図が作成されている（画像ファイルではなく）
- [ ] インラインコメントが「なぜ」を説明している
- [ ] リンク切れがない
