# Issue-Driven Development

## 概要

Issue駆動開発 (IDD) のワークフローガイド。全ての作業をGitHub Issueに紐付けることで、作業の追跡性、優先度管理、チーム間の透明性を確保する。DAG分解による大規模タスクの管理手法を含む。

## 適用場面

- 新機能の企画・実装計画
- バグ報告の管理と追跡
- 大規模リファクタリングの段階的実施
- チーム間のタスク調整

## ベストプラクティス

### 1. Issue作成

- タイトルは「何を」「どうする」を明確に記述する
- 本文にはコンテキスト、受入条件 (Acceptance Criteria)、完了定義を含める
- ラベルで種類を分類する: `feature`, `bug`, `refactor`, `docs`, `chore`
- 優先度ラベルを付与する: `P0`(緊急), `P1`(高), `P2`(中), `P3`(低)
- テンプレートを活用して品質の均一化を図る

### 2. DAG分解

- 大きなIssueは実装可能な粒度のサブタスクに分解する
- サブタスク間の依存関係をDAG (有向非巡回グラフ) として整理する
- 依存関係のないタスクは並列実行可能としてマークする
- 各サブタスクの見積もり時間を記録する
- Epic Issueにサブタスクのチェックリストをまとめる

### 3. ブランチとIssueの紐付け

- ブランチ名にIssue番号を含める: `feature/#123-add-auth`
- コミットメッセージにIssue番号を参照する: `feat: add login form (#123)`
- PRのdescriptionに `Closes #123` を記載して自動クローズを設定する
- 1 Issue = 1 PR を原則とする

### 4. ステータス管理

- `Open` -> `In Progress` -> `In Review` -> `Done` のフローに従う
- GitHub Projectsのボードでステータスを可視化する
- ブロッカーがある場合は即座にIssueコメントで共有する
- 長期間停滞しているIssueは定期的にトリアージする

### 5. 受入条件の定義

- 受入条件はテスト可能な形式で記述する
- Given-When-Then形式を推奨する
- UIの変更にはスクリーンショットやモックアップを添付する
- 非機能要件（パフォーマンス、セキュリティ）も受入条件に含める

## チェックリスト

- [ ] 全ての作業にIssueが紐付いている
- [ ] 受入条件が明確に定義されている
- [ ] ラベルと優先度が設定されている
- [ ] 大きなIssueはサブタスクに分解されている
- [ ] ブランチ名にIssue番号が含まれている
- [ ] PRにIssue参照が記載されている
